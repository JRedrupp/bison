# Contributing to bison

## Getting started

```bash
curl -fsSL https://pixi.sh/install.sh | sh
git clone https://github.com/JRedrupp/bison.git
cd bison
pixi install
```

Supported platforms: `linux-64`, `osx-arm64`. Requires `pixi >= 0.41.0`,
`max >= 26.2`.

## Development workflow

All tasks run through Pixi:

```bash
pixi run fmt            # auto-format with mojo format
pixi run check          # compile check (--Werror, zero warnings)
pixi run test           # run the full test suite
pixi run lint           # run all pre-commit hooks
```

Before submitting a PR, make sure `pixi run check` and `pixi run test` both
pass. The project enforces a zero-warnings policy.

## Implementing a stub method

Most contributions involve replacing a stub with a native Mojo implementation.

1. **Find a stub.** Look for any `_not_implemented(...)` call in `bison/`.
   GitHub issues #1-33 track each stub category.
2. **Implement it.** Replace the `_not_implemented(...)` call with native Mojo
   code. Remove the dummy `return self` / `return PythonObject(None)` if the
   real return type is different.
3. **Update tests.** In the corresponding `tests/test_*.mojo` file, remove the
   "expect raise" assertion and add real assertions comparing against pandas
   output. See [docs/testing.md](docs/testing.md) for the test file map.
4. **Submit a PR.** Reference the relevant GitHub issue in your PR description.
   CI will run lint, check, and test automatically.

### Stub pattern

Every unimplemented method must follow this exact form:

```mojo
fn sort_values(self, by: String, ascending: Bool = True) raises -> Self:
    _not_implemented("DataFrame.sort_values")
    return self   # never reached; satisfies type checker
```

Rules:
- The string passed to `_not_implemented` must be `"TypeName.method_name"`.
- Never use a bare `raise Error("not yet implemented")` — the pre-commit hook
  will block it.

## Code style

- **Language**: Mojo (`.mojo` files). Python is only used in `scripts/`.
- **Formatting**: `pixi run fmt` runs `mojo format`. This is enforced by
  pre-commit.
- **`def` not `fn`**: The `fn` keyword is deprecated on nightly Mojo. Use `def`
  for all new function and method definitions.
- **No emojis** in code or documentation.
- See [docs/mojo-patterns.md](docs/mojo-patterns.md) for Mojo-specific
  patterns and pitfalls.

## Architecture

See [docs/architecture.md](docs/architecture.md) for details on column storage,
type predicates, visitor dispatch, marrow integration, and adding new math
transforms.

Key rules:
- Never use `_data.isa[...]()` directly in `_frame.mojo` — use the `Column`
  predicate methods (`is_int()`, `is_float()`, etc.).
- After cache mutations, call `col._rebuild_marrow_only()` to sync backends.
- New element-wise math operations should follow the `_apply[F]` pattern
  documented in the architecture guide.

## Versioning

The version lives in exactly one place: `[project] version` in `pixi.toml`.
`bison/_version.mojo` is auto-generated by `scripts/gen_version.py` — never
edit it by hand.

## Project layout

```
bison/
├── bison/              — the Mojo package (import bison)
│   ├── __init__.mojo   — public API exports
│   ├── _errors.mojo    — _not_implemented() helper
│   ├── _version.mojo   — AUTO-GENERATED; never edit by hand
│   ├── _frame.mojo     — core DataFrame, Series, GroupBy
│   ├── column.mojo     — Column storage & ColumnData variant
│   ├── arrow.mojo      — Arrow <-> Column conversion (marrow interop)
│   ├── dtypes.mojo     — BisonDtype struct + dtype constants
│   ├── index.mojo      — Index, RangeIndex, ColumnIndex structs
│   ├── accessors/      — .str and .dt accessor structs
│   ├── io/             — read_csv, read_json, read_parquet, read_excel
│   └── reshape/        — concat
├── tests/              — one test_*.mojo file per feature area
├── benchmarks/         — performance benchmarks
├── scripts/            — gen_version.py, run_tests.sh, etc.
└── docs/               — documentation and performance dashboard
```
